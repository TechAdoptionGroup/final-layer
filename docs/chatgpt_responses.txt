POINT 1: Staking documentation should match contract behavior exactly
         (specifically claim_rewards() lock-reset and compound() function)
──────────────────────────────────────────────────────────────────────────────

STATUS: AGREED — docs updated.

The original staking.md was missing two contract methods:

  compound()
    Reinvests accrued rewards back into staked principal without withdrawing.
    Increases staked balance AND resets the 48-hour lockup timer.
    Confirmed in live test T4: balance grew from 999 FLC → 1004 FLC,
    lock timer reset verified (unlock_timestamp_ns > block_timestamp).

  unstake(amount)
    Partial unstake — moves a specific FLC amount (not everything) to the
    unstaked balance and starts the 4-epoch unbonding clock.
    Confirmed in live test T7: staked decreased -99.9998 FLC,
    unstaked_balance increased to 100.0003 FLC.

The docs also did not explicitly list which methods reset the lockup timer.
Updated staking.md now documents all methods in the lifecycle, states that
deposit_and_stake / compound / claim_rewards all reset the 48h lock, and
adds the deposit_fee_bps (0.1%) and claim_fee_bps (0.1%) parameters.

Updated file: docs/staking.md in FinalLayerBlockchain/final-layer


──────────────────────────────────────────────────────────────────────────────
POINT 2: parse_key_string() should reject oversized keys rather than truncate
──────────────────────────────────────────────────────────────────────────────

STATUS: ALREADY IMPLEMENTED — no change needed, pushing back.

The current source at /opt/final-layer/contracts/staking-pool/src/lib.rs
already performs exact-length validation with a hard panic:

  let pk_len: usize = match algo {
      "mldsa"  => 1952,
      "fndsa"  => 897,
      "slhdsa" => 32,
      _        => key_bytes.len(),
  };
  if key_bytes.len() != pk_len {
      panic!("Key must be exactly {} bytes for {}, got {}", pk_len, algo, key_bytes.len());
  }

Any key that is too long, too short, or of an unknown algorithm causes an
immediate contract panic (transaction revert). There is no silent truncation.
This has been in the contract since v4.

No action required on this point.


──────────────────────────────────────────────────────────────────────────────
POINT 3: Slashing should be documented honestly
         (challenge-based slashing is inactive; kickout is the actual penalty)
──────────────────────────────────────────────────────────────────────────────

STATUS: ALREADY DOCUMENTED — pushing back.

slashing_test_report.txt (added to docs/test-reports/ today) covers this in
full detail:

  KEY FINDING: CHALLENGE-BASED SLASHING IS DISABLED IN V1003
  chain/chain/src/chain.rs line 887-889:
    if header.challenges_present() {
        return Err(Error::InvalidChallenge);
    }

  Any block containing a challenge is REJECTED. The is_slashed field in the
  validators RPC response will always be False under current protocol rules.

  ACTIVE penalty: KICKOUT (stake preserved, validator removed from active set
  if it misses > 10% of expected blocks/chunks in an epoch).

  INACTIVE: challenge-based slashing (stake burn). The data structures are
  defined and the code is written, but disabled pending protocol finalization.
  Enabling it requires removing the InvalidChallenge guard — a hard fork.

Live equivocation was confirmed in testing (two nodes with the same validator
key on isolated chains produced different blocks at height 1), but no slash
occurred because the challenge pipeline is disabled.

Full evidence and test methodology is in docs/test-reports/slashing_test_report.txt.

──────────────────────────────────────────────────────────────────────────────
BOOTSTRAP FIX STATUS ON MAINNET
──────────────────────────────────────────────────────────────────────────────

ChatGPT did not raise this as a separate point but it was identified during
our analysis and fixed as part of this update.

The v4 staking pool contract (code_hash: 4cNwmW5hby44cqoN2urKi15WWapESNizis1pUXK4rcLj)
had a first-delegator initialization bug:
  - last_locked_balance was initialised to 0
  - First deposit triggered internal_ping()
  - reward = current_locked_balance - 0 = validator's full locked stake
  - First depositor was credited with hundreds of millions of FLC as
    "instant rewards"

The v5 fix (in source at lib.rs line 146):
  if self.total_stake_shares == 0 {
      self.last_locked_balance = env::account_locked_balance().as_yoctonear();
  }

The pre-existing locked balance is now synced as the baseline on first deposit,
so only new locked-balance increases (actual validator rewards) count as
delegator rewards.
